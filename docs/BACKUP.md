# Резервне копіювання БД (Supabase)

Нічний бекап усіх таблиць схеми `public` через GitHub Actions + ручне відновлення.

- **`.github/workflows/db-backup.yml`** — щоночі о 02:00 UTC знімає `pg_dump` (custom-формат, стиснення 9) + plain-SQL копію, зберігає артефактом GitHub, надсилає дамп документом у Telegram; щонеділі та при ручному запуску додатково відновлює дамп в ефемерний Postgres-контейнер (тест відновлюваності).
- **`.github/workflows/db-restore.yml`** — ручне відновлення в продакшн: повне або вибіркове по таблицях.

## Секрети

Settings → Secrets and variables → Actions → Repository secrets:

| Секрет | Обовʼязковість | Що це |
| --- | --- | --- |
| `SUPABASE_DB_URL` | обовʼязковий | Повний URI **Session pooler** з паролем БД |
| `TELEGRAM_BACKUP_BOT_TOKEN` | опційний | Токен бота для відправки дампа й сповіщень про збої (значення може бути те саме, що в наявному `TELEGRAM_BOT_TOKEN`, — окрема назва, щоб не зачіпати інші процеси) |
| `TELEGRAM_BACKUP_CHAT_ID` | опційний | ID чату, куди слати дамп і сповіщення |

### Де взяти `SUPABASE_DB_URL`

Supabase Dashboard → Connect → блок **Session pooler**:
хост виду `aws-0-<region>.pooler.supabase.com`, порт **5432**. У URI підставте пароль БД
(не API-ключ!). Спецсимволи в паролі — URL-екодувати (`@` → `%40` тощо).

Чому саме Session pooler:

- **Direct connection** (`db.<ref>.supabase.co:5432`) на Free-плані працює лише по IPv6, а runner'и GitHub Actions мають тільки IPv4 → connection timeout.
- **Transaction pooler** (порт 6543) несумісний із `pg_dump` (потрібен session-режим).

### Telegram

⚠️ Дамп містить **усі дані БД** (включно з персональними даними користувачів) —
`TELEGRAM_BACKUP_CHAT_ID` має бути **приватним** чатом (свій чат із ботом або
закрита адмін-група). Ліміт Bot API — 50 МБ на файл: більший дамп не надсилається
(зʼявиться warning у логах), беріть його з артефактів GitHub.

Формат повідомлення при успіху:

> ✅ Описовий Блукач: нічний бекап БД створено — 23 таблиць, дамп 4.2M (артефакт db-backup-2026-07-10).

При збої будь-якого кроку (включно з ротацією і тестом відновлення):

> ⚠️ Описовий Блукач: нічний бекап БД впав. \<посилання на запуск\>

## Як запустити вручну

Actions → **DB Backup** → Run workflow (кнопка зʼявляється лише після мержу workflow
у гілку `main` — з інших гілок не працює ні cron, ні кнопка). Ручний запуск примусово
виконує і тест відновлення (`verify-restore`).

Артефакти: сторінка запуску → секція **Artifacts** → `db-backup-YYYY-MM-DD`
(всередині `.dump` — custom-формат для `pg_restore`, `.sql.gz` — читабельна SQL-копія).

## Ротація артефактів

Після кожного успішного бекапу workflow лишає **10 найновіших** артефактів
`db-backup-*`, старіші видаляє через GitHub API (потрібен дозвіл `actions: write`,
уже прописаний у workflow). Запасний механізм — `retention-days: 14`: якщо ротація
не спрацює, GitHub сам зітре артефакти за 14 днів.

## Процедура відновлення

Actions → **DB Restore** → Run workflow:

1. `confirm` — введіть `RESTORE` (без цього job одразу падає).
2. `run_id` — ID запуску DB Backup, з якого брати дамп (порожньо = останній успішний).
3. `tables` — порожньо для **повного** відновлення; або список таблиць через кому для **вибіркового**.

**Повне відновлення** (`tables` порожнє): `pg_restore --clean --if-exists --single-transaction`
перестворює всі обʼєкти схеми `public` (таблиці, функції, RLS-політики) в одній
транзакції — при помилці все відкочується. `--clean` бере блокування на таблиці —
виконуйте в години низького трафіку (вночі, поза слотами розсилки бота).

**Вибіркове відновлення** (`tables` = `bot_users, confirmations`): перелічені таблиці
очищаються `TRUNCATE` **без CASCADE** — якщо на таблицю посилаються foreign key
з таблиць поза списком, крок впаде. Це навмисний запобіжник: додайте залежні
таблиці до списку. Потім дані заливаються з дампа (`pg_restore --data-only -t ...`).
У схемі проєкту немає тригерів-лічильників, тож перерахунки після заливки не потрібні.

## Ліміти безкоштовних планів і бюджет

| Ресурс | Ліміт | Наше споживання |
| --- | --- | --- |
| Supabase egress | 5 ГБ/міс | 1 прохід pg_dump за ніч (стиснений потік); plain-SQL і тест відновлення — з локального файлу, 0 egress |
| GitHub Actions (приватний репо) | 2000 хв/міс | бекап ~2–3 хв/ніч (~90 хв/міс) + тест відновлення раз на тиждень (~4×5 хв) |
| Сховище артефактів | 500 МБ | 10 артефактів × (дамп + sql.gz); при наближенні до ліміту — зменшити `KEEP` у кроці ротації або `retention-days` |

Бонус: нічний `pg_dump` — це активність для Supabase Free, проєкт не «засинає»
від неактивності. Якщо проєкт таки призупинять — дамп упаде і прийде сповіщення
в Telegram.

## Поза скоупом

- **Storage-бакети** Supabase не бекапляться: щонічне викачування файлів стало б основним джерелом egress. Схема `public` покриває всі дані застосунку (в `auth`/`storage` критичних даних немає — бот автентифікує через Telegram, веб-перевірка тримає користувачів у власних таблицях).

## Чек-лист після впровадження

1. Додати секрет `SUPABASE_DB_URL` (+ опційно Telegram-секрети).
2. Змержити workflow-файли в `main`.
3. Ручний запуск DB Backup — обидва job'и зелені, смоук-перевірка: N таблиць у дампі = N відновлено; у Telegram прийшов дамп із підписом.
4. Перевірити guard: запуск DB Restore без `RESTORE` у confirm має одразу впасти. Повне відновлення в прод заради тесту НЕ ганяти — `verify-restore` вже доводить відновлюваність.
