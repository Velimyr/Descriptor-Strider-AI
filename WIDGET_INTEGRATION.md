# Інтеграція віджета Блукач

Віджет «Блукач» дозволяє вашим відвідувачам допомогти розпізнати архівні справи прямо на вашому сайті. Один тег `<script>` — і користувачі бачать запрошення взяти участь, заповнюють кілька полів, отримують бали в спільному рейтингу.

## Швидкий старт

1. Напишіть нам, щоб ми створили вам акаунт партнера (див. [Реєстрація партнера](#реєстрація-партнера)). Ви отримаєте:
   - `partnerId` — короткий ідентифікатор (наприклад, `archium`).
   - `apiKey` — секретний ключ виду `blkch_…` (64 символи).

2. Додайте один тег в `<head>` або в кінець `<body>` вашого сайту:

   ```html
   <script
     src="https://blukach.app/widget.js"
     data-partner-key="blkch_ВАШ_КЛЮЧ_ТУТ"
     data-partner-id="archium"
     async
   ></script>
   ```

3. Готово. Відвідувачі побачать кнопку «Допомогти архіву» в правому нижньому куті.

## Як це виглядає для користувача

1. **Floater** — округла кнопка в кутку екрана з аватаркою Блукача й текстом «Допомогти архіву».
2. **Запрошення** — попап «Привіт! Я Блукач. Допоможи розпізнати кілька архівних справ — це займе хвилину.» з кнопками «Так, допоможу» / «Не зараз».
3. **Опитування** — фото фрагмента архівного опису + питання по черзі. Поля типу «Назва справи», «Роки», «Кількість аркушів». Кнопки «Назад» / «Не заповнено» / «Далі».
4. **Підтвердження** — зведення відповідей, кнопки «Підтвердити» / «Назад» / «Пропустити». Поряд із кожним полем є олівець для редагування.
5. **Результат** — «Дякую! Ви отримали +N балів. Сьогодні N справ.» Кнопка «Наступна справа» / «Закрити».

Для справ, які вже хтось розпізнав, користувач бачить **режим перевірки**: існуючі відповіді й кнопки «Усе правильно» / «Виправити» / «Пропустити».

## Підключення

### Мінімальний приклад

```html
<script
  src="https://blukach.app/widget.js"
  data-partner-key="blkch_…"
  data-partner-id="archium"
  async
></script>
```

### Атрибути

| Атрибут | Обовʼязковий | Опис |
|---|---|---|
| `src` | так | URL бандла віджета. Завжди `https://blukach.app/widget.js`. |
| `data-partner-key` | так | API-ключ партнера (видаємо ми). |
| `data-partner-id` | бажано | Короткий slug партнера (видаємо ми). Використовується для namespace в localStorage. Якщо не вказано — береться перші 12 символів ключа. |
| `data-api-base` | ні | Альтернативний backend (для тестів). За замовчуванням — origin самого `src`. |
| `async` | бажано | Не блокує рендер сторінки. |

### Куди вставляти

- В кінець `<body>` — найбезпечніше.
- В `<head>` з `async` — теж працює.
- Через GTM (Google Tag Manager) — підтримується, але переконайтесь що тег зʼявляється на всіх сторінках, де ви хочете показувати віджет.

### CSP (Content Security Policy)

Якщо у вас налаштований CSP:

```
script-src https://blukach.app;
connect-src https://blukach.app;
img-src https://blukach.app;
style-src 'unsafe-inline';  # потрібно, бо віджет інлайнить CSS у Shadow DOM
```

## Налаштування

На поточному етапі MVP кастомізація мінімальна — все керується через адмінку партнера, не через HTML-атрибути:

- **Назва префікса** в анонімному nickname (`Архіум-7f3a`) — встановлюється при реєстрації.
- **Allowed origins** — список доменів, з яких дозволено робити запити (точна звірка).

Плани на майбутнє (не в MVP):
- Налаштовування позиції кнопки (нижній лівий / правий кут).
- Власна кольорова схема.
- Власний текст запрошення.
- Можливість показувати тільки на певних сторінках.

## Реєстрація партнера

Зараз — через нашу ручну модерацію. Напишіть нам на пошту (адресу уточніть в основній команді) з інформацією:

1. **Назва організації** і коротко про вас (1-2 речення).
2. **URL сайту**, де планується встановити віджет. Якщо доменів декілька (наприклад, `archium.org` і `www.archium.org`) — вкажіть усі.
3. **Бажаний `partnerId`** (опційно) — латинськими літерами в kebab-case, наприклад `archium` або `local-history`.
4. **Префікс для анонімних nickname** — як ваші користувачі будуть зʼявлятись у рейтингу. Наприклад, «Архіум» → `Архіум-7f3a`.

Ми створюємо запис у системі і повертаємо вам:
- `partnerId`
- `apiKey` (показуємо один раз — збережіть у себе)
- Сніпет для копіювання

## Технічні деталі

### Як працює віджет

1. Скрипт завантажується асинхронно (≈65 КБ gzip).
2. Створюється елемент `<div id="blukach-widget-host">` у `<body>`.
3. Усередині — Shadow DOM з нашими стилями. Стилі вашого сайту не торкаються (і навпаки).
4. Юзер натискає floater → віджет робить `POST /session/start` → отримує анонімний токен сесії, який зберігається в `localStorage` під ключем `blkch:session:<partnerId>`.
5. Далі цей токен використовується для всіх запитів. Сесія живе 30 днів — повторні візити того ж браузера зараховуються тому ж нікнейму.

### Endpoints, які викликає віджет

| Метод | Шлях | Опис |
|---|---|---|
| POST | `/api/public/v1/session/start` | Створює анонімного юзера, повертає токен. |
| GET | `/api/public/v1/case/next` | Наступна справа для юзера. |
| POST | `/api/public/v1/case/:id/submit` | Відправити відповіді. |
| POST | `/api/public/v1/case/:id/confirm` | Підтвердити чужі відповіді без правок. |
| POST | `/api/public/v1/case/:id/skip` | Пропустити справу. |
| GET | `/api/public/v1/case/:id/image` | Фото справи (кешується CDN). |
| POST | `/api/public/v1/session/heartbeat` | Продовжує резервацію справи (тільки для collab-режиму). |
| GET | `/api/public/v1/me/stats` | Стата юзера. |

Кожен запит несе заголовок `X-Partner-Key: <ваш ключ>` і (де треба) `Authorization: Bearer <session_token>`. Усе це робить сам віджет — вам нічого не треба налаштовувати.

### Що зберігається в localStorage

Один ключ: `blkch:session:<partnerId>` зі значенням:

```json
{"sessionToken": "…", "userId": "web:…", "nickname": "Архіум-7f3a"}
```

Видалення цього ключа = новий анонімний юзер при наступному візиті.

## Безпека

### Що ми робимо

- **API-ключ зберігається як SHA256-хеш** у нашій БД. Plaintext віддаємо лише один раз при створенні.
- **Allowed origins** — точна перевірка походження запиту. Якщо хтось скопіює ваш ключ і вставить на інший домен — наш бекенд відмовить (403).
- **Session token** — підписаний HMAC. Підробити неможливо без знання серверного секрету.
- **CORS** — після валідації ключа і origin виставляємо `Access-Control-Allow-Origin: <ваш origin>`. Інші домени отримати дані не зможуть.
- **Зображення** — захищені окремим HMAC-токеном, прив'язаним до конкретної справи.

### Що варто памʼятати

- **Ключ потрапляє в HTML вашого сайту** (через `data-partner-key`) — це нормально. Він захищений origin-фільтром. Не публікуйте його в публічних репозиторіях окремо від origin-контексту.
- Якщо ключ скомпрометовано — напишіть нам, ми його ротуємо за 10 хвилин.
- **Анонімні юзери не реєструються** — це за дизайном. Хто хоче регулярно брати участь — переходить у наш Telegram-бот через посилання у віджеті.

## Питання та відповіді

**Чи можуть мої юзери побачити одну й ту саму справу двічі?**
Ні. Як тільки юзер взяв справу — їй проставляється «не давати цьому юзеру знову» в нашій БД.

**Скільки людей перевіряють одну справу?**
Залежить від режиму справи. Зазвичай — мінімум 3 (один розпізнає, ще двоє підтверджують).

**Що буде, якщо юзер закриє вкладку посеред відповіді?**
Якщо це справа в collaborative-режимі — резервація знімається через 30 хв або раніше через `sendBeacon` при закритті вкладки. У parallel-режимі — справа не блокується нікому.

**Чи можна вимкнути віджет на якихось сторінках?**
Зараз — лише не вставляючи скрипт на ці сторінки (через ваш CMS або GTM). Налаштовуваний whitelist URL — у планах.

**Чи можна побачити, скільки користувачів прийшло з мого сайту?**
Так, у нашій адмінці є статистика по партнеру (буде розширена).

**Чи отримуємо ми (партнер) копію розпізнаних даних?**
Ні, дані залишаються в проєкті Блукач і використовуються тільки для побудови архівних описів.
